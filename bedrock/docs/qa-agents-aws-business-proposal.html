Business and AWS architecture proposal

QA-Agents: Shared Verification Context trên AWS

Proposal này chuyển nội dung RFC thành một tài liệu dễ review: đang giải quyết vấn đề gì, cho ai, vì sao chọn hướng MVP hiện tại, các lựa chọn khác là gì, và AWS stack nên được dùng như thế nào khi triển khai từ Phase 1 đến Phase 2.

Tóm tắt quyết định

Mục tiêu không phải là tạo một agent tự động test thay QA. Mục tiêu là tạo một shared verification context để QA, Product và Engineer cùng nhìn thấy intent, risk, ambiguity, plan và evidence trước khi QA sign-off.

User chính

QA là primary user. Product và Engineer là context contributors khi cần xác nhận business rule, expected behavior hoặc implementation scope.

MVP artifact

Ready-for-QA Verification Context Record: artifact dùng trước khi QA execute để giảm sai scope và thiếu evidence.

AWS fit Phase 1

Bedrock/RAG + AgentCore Runtime/Harness boundary. AgentCore dùng đơn giản để có runtime, endpoint, session và trace.

Không làm ở MVP

Không auto pass/fail, không write ticket/doc tự động, không bắt buộc PR/source-code analysis, không triển khai full multi-tool agent.

Business problem đang giải quyết

QA không chỉ test. QA đang tổng hợp intent, business rule, current behavior, impact, test coverage, evidence và release memory từ nhiều nguồn rời rạc. Khi các nguồn này không cùng ngữ cảnh, QA dễ hiểu sai ticket, verify sai scope hoặc miss regression.

Pain cần xử lý

  • QA mất thời gian hiểu ticket.
  • QA verify sai scope vì ticket thiếu context.
  • Product, Engineer và QA không cùng một context.
  • Evidence sau QA thiếu hoặc viết muộn.
  • Regression bị miss do impact analysis yếu.

Business outputs

  • Shared verification context.
  • Risk summary.
  • QA sign-off note.
  • Release confidence record.
  • Defect-quality improvement.

Success metrics

  • Time-to-understand ticket.
  • Shared-context alignment.
  • Mismatch detection before testing.
  • Test coverage usefulness.
  • Evidence completeness and release confidence.

Options và rationale

RFC cần cho reviewer thấy không chỉ chọn gì, mà còn vì sao không chọn các hướng khác. Đây là phần đại diện cho business/product decision making.

Decision Options đã cân nhắc Option chọn Lý do chọn
Root problem Generate test cases, chatbot, shared context, autonomous QA agent Shared verification context Giải quyết root gap trước execution: ticket, business history, current behavior, impact và evidence bị phân tán.
Primary user QA-first, Product-first, Engineer-first, multi-role ngang nhau QA-first QA là người trực tiếp verify và tạo record; Product/Engineer contribute context khi cần.
MVP artifact Test matrix, clarification block, sign-off note, context record Ready-for-QA Verification Context Record Làm base cho test matrix, clarification, risk summary và final note.
Minimum context Ticket only, ticket + QA input, PR/source required, full RAG Ticket + QA input Đủ nhẹ để pilot; vẫn bắt QA nhập current behavior/history hoặc mark unknown để tránh AI bịa.
AWS fit Bedrock only, Bedrock + RAG, AgentCore boundary, full AgentCore agent Bedrock/RAG + AgentCore Runtime/Harness Dùng AgentCore đơn giản để có production-shaped boundary, chưa overbuild Gateway/write tools.
AI authority AI chỉ hỏi, AI trả lời có boundary, AI quyết định final AI trả lời có boundary Giảm clarification loop nhưng vẫn giữ human authority cho business rule, implementation và release risk.
Rule quan trọng: AI được trả lời khi có source/context rõ và phải gắn nhãn source-backed, inferred, hoặc unknown. AI không được pass/fail thay QA, không override Product/Engineer, và không biến inference thành fact.

AWS stack đề xuất

Phase 1 vẫn sử dụng AgentCore, nhưng ở mức đơn giản: runtime/harness boundary cho QA Assistant. Các stack như Lambda, SQS, EventBridge, Step Functions, DynamoDB, S3, CloudWatch và KMS xử lý phần app, async workflow, evidence, audit và vận hành.

Architecture overview

Diagram này thể hiện Phase 1 production-shaped architecture. AgentCore có mặt nhưng được giới hạn: host/wrap QA Assistant, không mở Gateway/write tools cho đến Phase 2.

Phase 1 boundary

Request vẫn đi qua AgentCore Runtime/Harness, nhưng agent chỉ tạo draft artifact. Tool calling được giới hạn ở retrieval đã được duyệt, không write external systems.

Phase 2 boundary

Khi product value được validate, thêm AgentCore Gateway, Identity, Policy và Memory để agent gọi ticketing, PR, CI, evidence storage và source-of-truth tools có approval.

Request journey trong Phase 1

Journey này giữ MVP gọn: input chỉ cần ticket + QA input. Retrieval là optional enrichment, không phải dependency bắt buộc.

Step Owner AWS stack Output
Input QA UI, Cognito, API Gateway Ticket + QA-entered current behavior/history/clarification.
Session System Lambda, AgentCore Runtime/Harness Validated request, runtime session, correlation id.
AI synthesis AI assistant Bedrock Converse, Guardrails, optional Knowledge Bases Context record with source-backed/inferred/unknown labels.
Review QA DynamoDB, S3, CloudWatch Approved record, clarification block, evidence checklist.
Escalation QA + Product/Engineer Manual in Phase 1; EventBridge/Gateway later Human confirmation for business/implementation/release risk.

Roadmap triển khai

Phase 0: RFC alignment

  • Chốt target user, pain, artifact, AWS fit.
  • Chốt data policy: PII, customer data, logs, secrets.
  • Chốt leading indicator cho pilot dashboard.

Phase 1: Simple AgentCore MVP

  • QA Workbench hoặc manual form.
  • AgentCore Runtime/Harness wraps assistant.
  • Bedrock prompt contracts + Guardrails.
  • DynamoDB/S3/CloudWatch for record and telemetry.

Phase 2: Tool-integrated agent

  • AgentCore Gateway exposes ticket/PR/evidence tools.
  • Identity/Policy control outbound access.
  • SQS/EventBridge/Step Functions handle async workflows.
  • AgentCore Evaluations measure agent/tool quality.
Layer Phase 1 scope Phase 2 expansion
AgentCore Runtime/Harness only; endpoint, session, trace discipline. Gateway, Memory, Identity, Policy, tool registry.
Bedrock Converse, prompt contracts, Guardrails, optional Knowledge Bases. Evaluation datasets, grounding checks, richer retrieval strategy.
Workflow Lambda sync request path; optional SQS for background export. Step Functions state machine for publish/review/escalation.
Integrations Manual copy/paste clarification, optional links only. Ticket comments, PR read tools, CI/test result tools, evidence uploader.
Governance Human review, labels, confidence/risk, no auto-write. Policy enforcement, approval UX, CloudTrail audit, custom KMS keys.
